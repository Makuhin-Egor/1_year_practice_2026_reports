\section{Постановка задачи}

Работа выполняется в рамках исследования открытых GPGPU архитектур на базе RISC-V ISA. 

\textbf{Целью работы является} освоение стека симуляции проекта Vortex, проведение параметрического анализа конфигураций и оценка эффективности масштабирования вычислительных ядер и кэш-иерархии с точки зрения производительности и потребления ресурсов ПЛИС.

Результаты работы необходимы для понимания архитектурных особенностей Vortex при выполнении задач, интенсивно работающих с памятью. Для исследователей архитектур и разработчиков это полезно тем, что позволяет избежать неэффективного использования ресурсов ПЛИС (LUT, BRAM) при конфигурации GPU для задач с поточным характером доступа к памяти. В выполнении работы заинтересована лаборатория/кафедра для формирования рекомендаций по настройке экземпляров Vortex под различные классы задач.

\section{Описание предлагаемого решения}

Для достижения цели был предложен подход, основанный на автоматизированном прогоночном тестировании модели Vortex с последующим извлечением метрик производительности и аналитической оценкой ресурсов ПЛИС. Идея решения заключается в варьировании двух главных архитектурных осей~--- количества ядер и размера кэш-иерархии~--- на задачах с различной арифметической интенсивностью: векторное сложение (\texttt{vecadd}), чувствительное к пропускной способности памяти, и умножение матриц (\texttt{sgemm}), имеющее высокую вычислительную нагрузку. Базовая конфигурация ядра включает 4 варпа (warp) и стандартный конвейер Vortex. Использование программного симулятора \texttt{simx} позволяет быстро оценивать поведение архитектуры без затрат на полный FPGA-синтез, при этом важно отметить, что \texttt{simx} не является строго потактово точным (cycle-accurate) по сравнению с RTL-симуляцией.

Реализация решения осуществлялась с использованием симулятора \texttt{simx} из состава Vortex, скриптов сборки Bash, языка Python для автоматизированного извлечения метрик из логов и визуализации данных (pandas, matplotlib), а также аналитической модели оценки ресурсов ПЛИС Xilinx. Для обеспечения воспроизводимости результатов был разработан пайплайн автоматизации, позволяющий запускать серию экспериментов одной командой.

\section{Эксперименты}

Экспериментальное исследование проводилось в три этапа: Strong Scaling (фиксированный объем данных, рост ядер), Cache Scaling (фиксированное число ядер, рост кэша) и Weak Scaling (пропорциональный рост данных и ядер). 

В качестве метрики производительности используется количество тактов симулятора (Cycles). Модель является детерминированной, поэтому результаты запусков воспроизводимы без статистического разброса. Для оценки ресурсов ПЛИС применялась аналитическая модель, где за единицу измерения принят стандартный блок блочной памяти Xilinx BRAM емкостью 4 КБ.

\textbf{Результаты Strong Scaling:}
Увеличение числа ядер с 1 до 4 на задаче \texttt{vecadd} (16K элементов) дало ускорение всего 1.05x. IPC каждого ядра упало с 0.79 до 0.21. Как видно на Рисунке~\ref*{fig:vecadd_strong}, реальное ускорение далеко от идеального. Ядра значительную часть времени простаивают в ожидании данных, что характерно для ситуаций, когда производительность ограничена подсистемой памяти. Данный эффект может быть связан как с аппаратными ограничениями пропускной способности памяти, так и с неоптимизированным размером рабочих групп (work-group size) при запуске OpenCL-ядра, что препятствует объединению запросов к памяти (memory coalescing).

\begin{figure}[H]
    \centering
    \includegraphics[width=0.7\textwidth]{images/1_strong_scaling_vecadd.png}
    \caption{Strong Scaling для задачи \texttt{vecadd}}
    \label{fig:vecadd_strong}
\end{figure}

Для сравнения, задача \texttt{sgemm} на 2 ядрах показала ускорение 1.33x, а на 4 ядрах~--- 1.51x (Рисунок~\ref*{fig:sgemm_strong}). Хотя \texttt{sgemm} имеет высокую арифметическую интенсивность, ускорение не достигает идеальных 4x. Это указывает на то, что объемов L1 кэша (16 КБ) недостаточно для полного покрытия рабочих множеств матриц, и ядра сталкиваются с задержками при обращении к подсистеме памяти.

\begin{figure}[H]
    \centering

    \caption{Strong Scaling для задачи \texttt{sgemm}}
    \label{fig:sgemm_strong}
\end{figure}

\textbf{Результаты Cache Scaling:}
Включение L2-кэша (1 МБ) для одного ядра на задаче \texttt{vecadd} дало +15\% ускорения, однако на 4 ядрах привело к \textbf{деградации} производительности. Добавление L3-кэша (2 МБ) не дало никакого прироста (Рисунок~\ref{fig:cache_impact}). Это объясняется характером доступа к памяти: \texttt{vecadd} обладает идеальной пространственной локальностью (последовательный обход массива), но полностью лишен временной локальности (данные читаются один раз и не переиспользуются). Следовательно, потоковые данные просто заполняют (pollute) большие объёмы L2/L3, не принося выигрыша от попаданий. Поскольку L1 кэш приватен для каждого ядра, разделяемый L2 выступает первой точкой сериализации доступа, где на множестве ядер возникает конкуренция за арбитраж (Contention) и конфликты по наборам кэша (Cache Set Conflicts) из-за выравнивания адресов, что и приводит к деградации. Для контраста, задача \texttt{sgemm} обладает высокой временной локальностью, однако её ускорение на 4 ядрах также ограничено задержками доступа к L2.

    \caption{Влияние иерархии кэшей на время выполнения (Cycles) для 1 и 4 ядер}

\end{figure}

\textbf{Результаты Weak Scaling (задача \texttt{vecadd}):}
При пропорциональном увеличении объема данных с ростом числа ядер (16K/1 ядро, 32K/2 ядра, 64K/4 ядра) время выполнения росло почти линейно: от 622 тыс. тактов для 1 ядра до 2.38 млн тактов для 4 ядер. В идеальной архитектуре время должно оставаться постоянным. Как видно на Рисунке~\ref*{fig:weak_scaling}, подсистема памяти является узким местом, создавая конкуренцию при параллельном доступе, даже при отсутствии пересечения данных между ядрами.

    \centering
    \includegraphics[width=0.7\textwidth]{images/4_weak_scaling.png}
    \caption{Weak Scaling: рост времени выполнения при увеличении объема работы}

\end{figure}

\textbf{Оценка ресурсов ПЛИС (Аппаратный ROI):}
В связи с отсутствием синтеза для целевой ПЛИС, оценка потребления ресурсов выполнена аналитически на основе объемов памяти конфигураций. Базовая конфигурация (1 ядро, 16 КБ L1 I\$ + 16 КБ L1 D\$) требует 8 BRAM. Конфигурация (4 ядра + L2 + L3) требует: 4 ядра $\times$ 8 BRAM (L1) + 256 BRAM (L2, 1 МБ) + 512 BRAM (L3, 2 МБ) = 800 BRAM. При этом данная конфигурация не обеспечивает прироста скорости для задач с поточным доступом, что демонстрирует снижение эффективности использования ресурсов ПЛИС (Speedup / BRAM) при инвестициях в кэш-иерархию без учета паттернов доступа и оптимизации программной модели.

\textbf{Выводы из эксперимента:} Масштабирование вычислительных ядер ограничено для задач, чувствительных к пропускной способности памяти, при неоптимизированных параметрах запуска OpenCL-ядер. Увеличение объема кэш-иерархии (L2/L3) неэффективно для задач с нулевой временной локальностью: это приводит к заполнению кэша без выигрыша от попаданий и конкуренции за порты. Полные таблицы с результатами замеров доступны в репозитории.

\section{Заключение}

В ходе выполнения работы получены следующие результаты:

\begin{itemize}
    \item Освоен стек симуляции и сборки проекта Vortex GPGPU.
    \item Разработан автоматизированный пайплайн (Bash + Python) для извлечения метрик из логов и визуализации результатов, обеспечивающий воспроизводимость исследования.
    \item В ходе Strong Scaling экспериментов при выбранных конфигурациях наблюдалось ограничение масштабирования для задач, чувствительных к пропускной способности памяти (\texttt{vecadd}, ускорение 1.05x на 4 ядрах), на фоне более эффективного ускорения задач с высокой арифметической интенсивностью (\texttt{sgemm}, 1.51x). Данное поведение может быть обусловлено ограничениями пропускной способности подсистемы памяти в сочетании с неоптимизированными параметрами запуска OpenCL-ядер (размер рабочих групп).
    \item Показано, что при выбранных параметрах запуска увеличение объема кэш-иерархии (L2/L3) не приводит к росту производительности для задач с нулевой временной локальностью: это вызывает заполнение кэша без выигрыша от попаданий и конкуренцию за порты (Contention).
    \item Выполнена аналитическая оценка потребления ресурсов ПЛИС (BRAM), показавшая снижение аппаратного ROI конфигураций с большими кэшами для задач потокового типа.
\end{itemize}
